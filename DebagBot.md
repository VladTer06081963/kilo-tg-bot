# Анализ безопасности и стабильности telegram-kilo-bot

В этом документе описаны найденные проблемы в файле `bot.js` и предложены пути их решения для улучшения работоспособности, защиты и устойчивости проекта.

## 1. Безопасность (Security) 🚨

Это наиболее критичный раздел. В текущей реализации бот уязвим к нескольким серьезным атакам.

### 1.1. Отсутствие авторизации (Access Control)

   **Проблема:** В данный момент любой пользователь Telegram, нашедший бота, может взаимодействовать с ним: создавать директории, удалять проекты и запускать генерацию через Kilo CLI. Это может привести к исчерпанию лимитов API, засорению диска и выполнению несанкционированных действий на сервере.

   **Решение:** Добавить проверку по белому списку (Whitelist).

* В `.env` файле указать ID разрешенных пользователей (например, `ADMIN_IDS=123456789,987654321`).

* В коде (через middleware или в начале каждого обработчика команд) проверять: `if (!ALLOWED_IDS.includes(msg.from.id)) return;`.

### 1.2. Уязвимость к инъекции команд (Command Injection)

*
   **Проблема:** В функции `/run` используется `exec` для запуска команды: `await runCommand('${KILO_PATH} run --auto "${command}"', chatId);`. Если пользователь передаст специально сформированную строку (например, `запрос" && rm -rf / && echo "`), это выполнится как Bash-скрипт на сервере.
*
   **Решение:** Использовать `child_process.spawn` вместо `exec`, передавая аргументы массивом, что предотвращает интерпретацию shell-символов.
 Или использовать `exec` с `{ shell: false }`, но это не всегда надежно.

* Либо надежно экранировать кавычки и спецсимволы в переменной `command` перед передачей в `exec`.

### 1.3. Уязвимость к Path Traversal (чтение/удаление чужих папок)

    **Проблема:** В команде `/create` имя проекта очищается (`replace(/[^a-zA-Z0-9-_]/g, '')`), а в команде `/delete` — нет (`const projectName = match[1].trim()`). Если передать боту команду `/delete ../../../.ssh`, он попытается удалить папку `~/.ssh`.

    **Решение:**

    * Применять такую же строгую валидацию (регулярным выражением) для имени проекта в команде `/delete`.
    * Дополнительно проверять, что итоговый (resolved) путь действительно находится внутри `PROJECTS_DIR` (например, с помощью `path.resolve` и проверки префикса).

## 2. Устойчивость и стабильность (Stability) 🛠

### 2.1. Падение Node.js из-за неперехваченных исключений (Unhandled Exceptions)

* **Проблема:** Использование синхронных методов `fs`, таких как `fs.rmSync` в `/delete`. Если папка заблокирована, отсутствуют права доступа или внутри есть защищенные файлы, `fs.rmSync` выбросит исключение. Поскольку блока `try...catch` нет, процесс Node.js (и сам бот) упадет. Синхронные методы также блокируют Event Loop, что плохо для масштабируемости.
* **Решение:**
* Обернуть вызовы `fs` (особенно удаление) в блок `try { ... } catch (e) { ... }`.
* Выводить сообщение об ошибке пользователю в Telegram, вместо падения приложения.
* Рекомендуется переход на асинхронные методы (например, `fs.promises.rm`).

### 2.2. Переполнение буфера `exec` (MaxBuffer Exceeded)

* **Проблема:** Функция `exec` имеет встроенное ограничение на объём вывода (по умолчанию 1 МБ). Если команда (например, Kilo CLI или `npm install`) сгенерирует слишком много логов, процесс упадет с ошибкой `ERR_CHILD_PROCESS_STDIO_MAXBUFFER`.
* **Решение:** Увеличить лимит в объекте настроек `exec`: `const execOptions = { maxBuffer: 1024 * 1024 * 5, ... };` (здесь 5 МБ).

### 2.3. Превышение лимита символов Telegram (4096 символов)

* **Проблема:** Функция форматирования вывода ограничивает `stdout` до 3500 символов, а `stderr` до 500. Вместе со служебными символами формата Markdown (`` ``` ``) и дополнительным текстом, итоговое сообщение может превысить жесткий лимит Telegram API в 4096 символов. Это вызовет ошибку отправки сообщения и `Unhandled Promise Rejection`.
* **Решение:**
  * Уменьшить лимиты (например, `stdout` до 3000, `stderr` до 200).
  * Добавить обработчик `.catch()` к `bot.sendMessage`.
  * Для слишком длинных логов лучше сохранять их во временный файл и отправлять пользователю как документ (`sendDocument`).

## 3. Логика и неочевидные баги 💡

### 3.1. Хардкод пути к Node.js

* **Проблема:** В `runCommand` жестко задан путь к бинарникам (включая путь NVM: `/Users/vlad/.nvm/versions/node/v20.18.0/bin`). При обновлении Node.js или переносе проекта на другой сервер (с другим именем пользователя), команды `npm` и `node` перестанут работать.
* **Решение:** Динамически формировать `PATH` на основе текущего окружения (`process.env.PATH`), либо вынести эти системно-зависимые пути в файл конфигурации (`.env`).

### 3.2. Некорректная реализация команды `/status`

* **Проблема:** В обработчике `/status` вызывается команда `exec('kilo --version')`. Однако, если глобально `kilo` не установлен, команда вернет ошибку, хотя бот успешно использует абсолютный путь из `KILO_PATH`. В итоге статус будет ложным.
* **Решение:** Использовать переменную окружения для вызова в `/status`: `exec(`"${KILO_PATH}" --version`, ...)`.

## Рекомендации по исправлению

1. **Срочно:** Реализовать проверку Telegram ID пользователя во всех командах для предотвращения доступа посторонних.
2. **Срочно:** Исправить уязвимость Path Traversal в команде `/delete` путем валидации передаваемого имени проекта.
3. Обернуть синхронные файловые операции в `try...catch`.
4. Заменить вызов `exec` в команде `/run` на более безопасный подход.
